A methodology and spec format for bundling AI-generated software specs with tiered, automated acceptance tests, so that 'the code compiles' stops being mistaken for 'the software actually works.'

Problem: AI coding assistants are good at producing code that compiles/builds from a spec, but bad at end-to-end validation of the running result. Concrete failure modes: Terraform code compiles and terraform plan looks fine, but terraform apply fails against real prod state (drift, existing resources, quotas); gcloud/GCP automation code is syntactically correct but fails to authenticate or is blocked by an org policy that only manifests at runtime against a real org; a web app builds and deploys but login is broken or the UI silently shows zero data once you actually click through it.

Core proposal: every feature-level spec (SPEC.md) should ship bundled with acceptance tests, not just prose 'Goals'/'Acceptance Criteria' sections. Three tiers of validation, chosen deliberately for cost/speed tradeoffs:
1. Deterministic fixtures/playgrounds: reproducible, semi-prod-like environments the coding AI can request or that are provisioned for it up front, e.g. 'give me an authenticated gcloud project to test this script against', a seeded staging database, a sandboxed cloud project with known quotas/policies. These make tier 2 and 3 possible by giving them something real to run against instead of mocks.
2. Unit/integration tests: traditional, cheap, deterministic, run on every iteration (hundreds of times) against the tier-1 fixtures where relevant.
3. LLM-as-judge end-to-end evals: for fuzzy, hard-to-assert correctness (does the webapp actually let you log in and see real data; does the terraform apply leave the intended resources in the intended state), a small number of automated runs where an agent actually drives the real system (e.g. Playwright clicking through the deployed web app, authenticated gcloud calls validating actual resource state after an apply) and an LLM evaluates whether the observed behavior matches intent. Optionally captured as video/screenshot evidence for human review. These are expensive and slow, so they run sparingly (e.g. once per milestone or pre-merge), not on every change like tier 2.

Explicitly OUT OF SCOPE for this spec: orchestrating multiple independent parallel implementation attempts of the same spec and comparing/judging between them (e.g. running the same spec through an agentic coding tool N times and picking the best result via video review). That is a separate, later concern; this spec is only about the format/methodology for bundling tiered acceptance tests with a spec and how those tests get authored and executed, not about multi-attempt orchestration.

Goal: define what a 'spec bundled with tests' looks like (what sections/artifacts it needs beyond prose), how a coding AI should be instructed to use it (write the tier-1 fixture requirements and tier-2/3 tests as part of implementing the feature, not as an afterthought), and how the tiers trade off cost vs confidence so cheap tests run constantly and expensive LLM/video evals run only where warranted.
